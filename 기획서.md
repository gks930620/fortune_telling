# 오늘의 운세 — 기획서

> AI가 정해진 시각에 **모든 경우의 수**의 운세를 미리 생성해 정적 사이트로 게시하고,
> 사용자는 자기 정보(생년월일·혈액형·MBTI)를 입력하면 **이미 준비된 결과를 즉시** 보는 서비스.
>
> 설계 근거: `자동화프로젝트 주제찾기/AI-정기실행-자동화-설계패턴.md`
> (코드와 AI의 일을 자른다 · AI를 역할별로 쪼갠다 · 파일이 기억이다)

---

## 1. 핵심 컨셉

- **요청 시 AI 호출 = 0회.** AI는 매일 새벽 배치에서만 돌고, 사이트는 100% 정적(무료, 서버 0원).
- AI는 개인별이 아니라 **버킷(경우의 수)별**로 생성한다.
  예: 띠 운세는 "1990년생용"이 아니라 **12개 띠 각각** 하나씩. 사용자의 출생연도 → 띠 변환은
  브라우저의 **코드**가 한다. (연도별로 준비하는 게 아니라 12종만 준비하면 모든 연도가 커버됨)
- 운세 텍스트(판단·해석·글)는 AI가, 날짜·간지·띠·별자리 **계산은 전부 코드**가 한다.

## 2. 운세 6종과 경우의 수(버킷)

| # | 운세 | 버킷 수 | 사용자 입력 → 버킷 변환 (코드) |
|---|---|---|---|
| 1 | 띠 운세 | **12** (쥐~돼지) | 생년월일 → 입춘(2/4) 기준 띠 판정 |
| 2 | 사주 운세 | **10** (일간 갑~계) | 생년월일 → 일주 60갑자 계산 → 일간 10종. 오늘의 일진과의 관계로 해석 |
| 3 | 별자리 운세 | **12** (물병~염소) | 생일 월·일 → 별자리 |
| 4 | 타로 | **22** (메이저 아르카나) | 입력 없음 — 아래 §3 참고 |
| 5 | 혈액형 운세 | **4** (A·B·O·AB) | 직접 선택 |
| 6 | MBTI 운세 | **16** | 직접 선택 |

**합계: 하루 76버킷.** 사주는 일간(10종) 기준으로 단순화한다 — 출생 "시각"은 받지 않는다.
(정밀화가 필요하면 나중에 60일주로 확장 가능. 버킷 수만 60으로 늘면 됨.)

## 3. 타로 동작 방식 (질문에 대한 답)

- AI가 매일 **22장 전부**에 대해 "그 카드가 오늘 나왔을 때의 해석"을 미리 생성한다.
- 사용자는 화면에서 **뒷면으로 섞인 22장 중 1장을 직접 선택**한다 (선택 = 사용자의 랜덤).
- 즉, "모두에게 똑같은 카드"가 아니라 **누가 어떤 카드를 뽑을지만 랜덤**이고,
  같은 날 같은 카드를 뽑은 사람은 같은 해석을 본다. → 사전 생성 원칙과 완벽 호환.

## 4. 발행 주기와 스케줄

실행은 **하루 1번 단일 cron**. 스크립트가 KST 날짜를 보고 그날 만들 주기를 스스로 판정한다.

| 주기 | 발행 조건 (KST 기준) | 내용 |
|---|---|---|
| daily | 매일 | 오늘의 운세 (76버킷) |
| weekly | 월요일 | 이번 주 흐름 (76버킷, 주간 톤) |
| monthly | 매월 1일 | 이번 달 운세 |
| yearly | 1월 1일 | 올해 운세 |
| holiday | 공휴일 테이블 매칭 시 | 연휴 특집 (설날=신년·귀성운, 추석=가족·재물운 등) |

- cron: `30 20 * * *` (UTC) = **KST 05:30**. GitHub Actions cron은 수십 분 지연될 수 있어
  "아침 6시에 보면 준비돼 있음"을 보장하려고 30분 여유를 둔다.
- 겹치는 날(예: 1/1 = daily+monthly+yearly+신정)은 해당 주기를 **전부** 생성한다.
- 공휴일 테이블: `scripts/holidays.js`에 2026~2027 하드코딩(대체공휴일 포함, 웹 검색으로 검증).
  **매년 말에 다음 해를 추가하는 것이 유일한 정기 유지보수 작업**이다. 테이블에 없는 연도가 오면
  스크립트가 경고 로그를 남긴다.

## 5. 파이프라인 (매 실행)

```
[cron 05:30 KST]
  ① context.js  (코드)  KST 날짜·요일, 오늘의 일진(60갑자), 연도 간지, 발행 주기 판정,
                        연휴 매칭, 다가오는 공휴일 → work/<날짜>/context.json 커밋(스냅샷)
  ② 역할 AI 6명 (독립 세션, 서로 모름)
       띠 전문가 → work/<날짜>/tti.json
       사주 명리가 → saju.json     별자리 점성가 → zodiac.json
       타로 리더 → tarot.json      혈액형 → blood.json      MBTI → mbti.json
       각자 "자기 파일 하나만" 생성. git 금지. 스키마는 prompts/common.md에 고정.
  ③ writer AI 1명  ②의 JSON들을 읽고 사람이 읽을 리포트 집필
       → docs/_posts/<날짜>-<주기>.md (주기별 1개)
  ④ assemble.js (코드)  ②출력 스키마 검증(버킷 키 완전성)·병합
       → docs/data/<주기>/<날짜>.json + docs/data/latest.json 갱신
  ⑤ git commit·push (워크플로가) → GitHub Pages(Jekyll)가 자동 재빌드
```

- **역할 AI에게 산수 금지**: 일진·간지는 ①이 계산해 context.json으로 주입한다. AI는 해석만.
- **한 역할 실패 ≠ 전체 실패**: 역할별 `|| true`, assemble은 없는 타입을 건너뛰고 로그.
  사이트는 빠진 운세에 "오늘은 준비 중" 표시. writer는 존재하는 파일만 종합.
- **멱등성**: 같은 날짜 재실행 시 전부 덮어쓰기 → 안전. `FORTUNE_DATE=YYYY-MM-DD`로
  날짜 오버라이드(재생/백필) 가능.

## 6. AI 역할·모델 설계

| 세션 | 인원 | 모델/강도 | 근거 |
|---|---|---|---|
| 역할 AI (6종) | 6 | sonnet / 기본 | 정형 스키마 채우기 — 다양성보다 물량 |
| writer | 1 | opus / high | 사람이 읽는 최종 글 — 품질 우선 |

- 인증은 **구독 토큰(`CLAUDE_CODE_OAUTH_TOKEN`)만** 사용. API 키를 환경에 두지 않는다(과금 방지).
- 하루 비용 = 7세션 × 1회. 연휴·월초 등 겹치는 날만 주기 수만큼 출력이 길어진다.
- 프롬프트는 `prompts/`에 파일로 두고 버전 관리. 실행 시 `역할.md + common.md + [실행 안내]` 조합 주입.
  날짜·일진·출력 경로는 워크플로가 주입한다(AI는 오늘이 며칠인지 모른다).

## 7. 데이터 스키마 (기계용)

역할 출력 `work/<날짜>/<type>.json` — 키는 코드와 프롬프트가 공유하는 **고정 ID**:

```json
{
  "type": "tti",
  "date": "2026-07-20",
  "periods": {
    "daily": {
      "rat":   { "score": 4, "keyword": "재정비", "text": "…2~3문장…", "advice": "…한 문장…", "lucky": { "color": "남색", "number": 3 } },
      "ox":    { "…": "…" }
    }
  }
}
```

- 버킷 ID — 띠: `rat ox tiger rabbit dragon snake horse sheep monkey rooster dog pig` /
  사주 일간: `gap eul byeong jeong mu gi gyeong sin im gye` / 별자리: `aries…pisces` /
  타로: `"0"~"21"` / 혈액형: `A B O AB` / MBTI: 16개 대문자.
- `score` 1~5 정수. assemble.js가 **키 완전성·타입을 검증**하고 불량이면 그 타입만 제외.

병합 결과(사이트용) `docs/data/daily/2026-07-20.json`:

```json
{ "date": "2026-07-20", "period": "daily",
  "context": { "day_ganzhi": "을미", "year_ganzhi": "병오", "weekday": "월" },
  "tti": { "rat": {…}, … }, "saju": {…}, "zodiac": {…}, "tarot": {…}, "blood": {…}, "mbti": {…} }
```

`docs/data/latest.json` = 주기별 최신 발행 날짜 포인터(클라이언트 폴백용).

## 8. 사이트 (Jekyll · GitHub Pages · `docs/` 루트)

- **index** — 생년월일 입력(+혈액형·MBTI 선택) → localStorage 저장 → 다음 방문부터 즉시 내 운세.
  띠·일간·별자리 계산은 `assets/js/manse.js`(브라우저)가 수행 — `scripts/lib/manse.js`와 동일 로직.
  타로는 22장 카드 선택 UI. 오늘 데이터는 `data/daily/<KST오늘>.json`, 없으면 latest로 폴백.
- **리포트 아카이브** — writer가 쓴 md가 `docs/_posts/`에 쌓여 Jekyll 포스트로 게시(주기별 필터).
- 내부 파일(prompts/, scripts/, work/)은 `docs/` 밖이라 빌드에 안 섞인다
  (프론트매터 오인 사고 방지 — 설계패턴 §3.7).

## 9. 저장소 구조

```
오늘의 운세/
├─ 기획서.md  README.md
├─ .github/workflows/fortune.yml   ← cron 30 20 * * * (UTC) = KST 05:30
├─ prompts/    common.md + role-{tti,saju,zodiac,tarot,blood,mbti}.md + writer.md
├─ scripts/    lib/manse.js  holidays.js  context.js  assemble.js   (의존성 0, Node 표준만)
├─ work/<날짜>/  context.json + 역할 JSON 6개   ← 원본 스냅샷(검증 근거, 커밋됨)
└─ docs/       Jekyll 루트 (GitHub Pages)
   ├─ _config.yml  _layouts/  index.html  reports.html  assets/
   ├─ _posts/<날짜>-<주기>.md          ← writer 산출물
   └─ data/{daily,weekly,monthly,yearly,holiday}/<날짜>.json + latest.json
```
